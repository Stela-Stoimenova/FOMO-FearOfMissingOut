BACKEND (5 layers)
1.Routes: define URLs like /api/events/:id/tickets
2.Middleware: checks JWT and role (RBAC)
RBAC - Role Based Access Control is a policy-neutral access control mechanism defined around roles and privileges
3. Controller: reads req, calls service, returns res 
4.Service: business logic (commission, loyalty pricing) + Prisma DB actions
5.Database: Prisma schema + migrations +contraints (unique ticket)

Every API call goes through the same pipline
Route -> Middleware -> Controller -> Service -> Prisma (DB) -> Controller -> Response 
Controllers are thiin. They validate request flow and return HTTP responses, while services contains business logic and database operations
req.user.userId - JWT payload includes userId, and middleware stores it in req.user
RBAC is enforced in the routes layer using middleware that checks JWT and allowed roles.
Auth endpoints delegate to an auth service that handles hashing and token generation.

“PostgreSQL is a relational database suitable for transactional systems. Since ticket purchase requires consistency and atomic transactions, a relational model with constraints is appropriate.”

“Prisma ORM provides type-safe database queries and migration management, reducing SQL injection risk and improving maintainability.”

“JWT allows stateless authentication, which simplifies horizontal scaling and removes the need for server-side session storage.”

“To preserve economic consistency. If the event price changes later, historical transactions must remain accurate.”

Password hashing is necessary so that raw passwords are never stored in the database. If the database is compromised, attackers cannot directly read user passwords. Bcrypt also uses salting, which protects against rainbow table attacks.

Without a database transaction, the system could create a ticket but fail to create the commission record or loyalty update. That would leave the database in an inconsistent state. Using $transaction guarantees atomicity: either all operations succeed, or none are applied.

The unique constraint enforces a business rule at the database level. Even if application logic fails, the database itself prevents duplicate ticket purchases for the same event by the same user.

Database-level constraints are stronger than application logic.

The frontend cannot be trusted because users can modify client-side code. Authorization must be enforced on the backend to prevent privilege escalation or malicious requests.
